# Plan: Fix Telegram Plugin Discovery + Sync Upstream

## Context

There are two interrelated issues with the current build. First, **all 70+ extensions** have a plugin id mismatch between their `package.json` name (e.g. `"ava-claw-telegram"`) and their `avaclaw.plugin.json` manifest id (e.g. `"telegram"`). The plugin discovery code in `src/plugins/discovery.ts` uses the package name as the "entry hint" id, which doesn't match the manifest id — producing warnings for every single plugin. Second, when the source `extensions/` directory gets scanned (dev mode, stale build, or path misconfiguration), the `./index.ts` entry in `package.json` triggers a boundary check failure because `openBoundaryFileSync` runs against a `.ts` file that may not resolve cleanly through the safe-open pipeline.

The upstream repo (`mia-openclaw`) has diverged significantly — **9,894 files changed, 850K insertions**. This is far too large for cherry-picking; it needs a proper `git merge`. The upstream fixes include: restructured plugin SDK imports, Telegram token resolution for binding-created accountIds, webhook rate limiting, queue-based thread binding persistence, and new module entry points.

**Strategy**: Phase 1 fixes the immediate blocker (extension naming + rebuild). Phase 2 does the full upstream merge separately.

| Current State                                    | Target State                                    |
| ------------------------------------------------ | ----------------------------------------------- |
| 70+ plugin id mismatch warnings                  | Clean startup with zero warnings                |
| `./index.ts` escape error in dev/stale scenarios | Entries resolve cleanly in both source and dist |
| Version 2026.4.1                                 | Version 2026.4.2 (patch)                        |
| 850K lines behind upstream                       | Merged and adapted                              |
